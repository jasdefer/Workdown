End-to-end smoke test of the build and serve pipeline: `workdown serve` boots axum, the Vite-built Svelte bundle is embedded via `rust-embed`, the browser loads a placeholder page. No real API, no real UI — just proves the plumbing all the way from `cargo build` to a working page in the browser.

## Architecture decisions (settled in design discussion)

- **Single binary, `serve` is a subcommand.** `workdown serve` lives on the existing `workdown` CLI, not as a separate binary. `crates/server` stays a library; the CLI's `main.rs` calls into it when `serve` fires. Preserves ADR-006's single-binary property — `cargo install workdown`, the cargo-dist PowerShell/shell installers, package-manager pipelines all keep working unchanged.
- **Stack: Svelte + TypeScript + Vite + rust-embed + axum.** Confirms ADR-006. Considered and rejected:
  - **htmx + server-rendered HTML.** Would remove the node toolchain and the JSON API. Rejected: prefer the SPA model — HTML lives in HTML files, data flows over JSON, frontend stays typed end-to-end.
  - **C# (Razor Pages / Blazor) calling the Rust CLI.** Would add a .NET runtime requirement (breaks the single-binary distribution), require two-language maintenance, and either re-implement core in C# or shell out per request. Rejected.
- **`rust-embed` with `debug-embed = false`.** Debug builds read `ui/dist/` from disk at runtime; release builds bake it in. Plain `cargo build` / `cargo check` / `cargo test` during the dev loop stay pure-Rust and fast — only release builds need the JS bundle on disk. UI iteration loop: Vite dev server (`npm run dev`) with HMR, `cargo run -- serve` for the backend, Vite proxies `/api/*` to the backend port.
- **Server returns JSON; UI renders.** The "API" is an internal contract between the two halves of the same binary (per ADR-006) — no public-stability guarantee, only consumer is the bundled Svelte UI.
- **Build pipeline plumbing: `xtask`.** A workspace member `xtask` (regular Rust binary) orchestrates the multi-step release build: `npm run build` then `cargo build --release`. Invoked as `cargo xtask build` (via `.cargo/config.toml` alias). Keeps plain `cargo build` / `cargo check` / `cargo test` pure-Rust so the dev loop never touches node. Considered and rejected: `build.rs` shelling out to npm (rebuild-cache fragility, runs npm on every target in cargo-dist's per-target matrix, fights the `debug-embed = false` design); hybrid `build.rs` gated by feature flag (collapses into "xtask plus a build.rs").
- **cargo-dist integration: per-target npm build via `github-build-setup`.** cargo-dist's `github-build-setup` field in `dist-workspace.toml` points at a YAML snippet (`.github/build-setup.yml`) with `setup-node` + `npm ci` + `npm run build` steps that get spliced into every per-target build job before `cargo build` runs. Each target VM builds its own `ui/dist/`. Considered and rejected (for now): build-once-and-fan-out via a shared artifact — better reproducibility and ~3 min faster per release, but requires fighting cargo-dist's autogenerated `release.yml` (no first-class "shared pre-build artifact" feature). Acceptable to revisit if release time becomes painful or we add many more targets.
- **Frontend stack: SvelteKit + `adapter-static` in SPA-fallback mode.** Scaffolded via `npm create svelte@latest ui` (TypeScript, `adapter-static`). Configured with `fallback: 'index.html'` so unknown URLs are handled by the client-side router. Output goes to `ui/dist/`. axum serves `index.html` for `/` and any unknown route, plus the JS/CSS bundle at SvelteKit's `/_app/*` paths. The SvelteKit server runtime (server hooks, form actions, `+page.server.ts`) is compiled out — only static assets ship in the binary. Considered and rejected: plain Svelte + Vite + a third-party router — smaller surface area, but the Svelte ecosystem has consolidated around SvelteKit (tutorials, examples, official path), and file-system routing pays off across the multiple view types in scope (board, tree, graph, table, gantt, charts).
- **CSS framework: deferred.** Skeleton uses plain CSS / Svelte scoped styles. Pick a styling system in slice 2 (`first-view-end-to-end`) when real UI design starts; retrofit is one command (`npx svelte-add tailwindcss` or equivalent).
- **Port behavior: default 3141, configurable, scan on conflict.** Override hierarchy: `--port N` flag > `.workdown/config.yaml` `serve.port` > default 3141. If the resolved port is taken, scan upward up to 10 ports (3141 → 3150) and log a warning showing the fallback. If `--port N` was explicit, hard-fail rather than scanning (user said *that* port). Startup log always prints the URL clearly. 3141 chosen as memorable (first four digits of π) and not commonly claimed by other dev tools. Multi-project workflow handled via `serve.port` in committed project config, not via opaque hashing — the repo decides its URL, contributors get the same one.
- **`--open` flag: opt-in, handled in `cli` crate.** `workdown serve` does not open a browser by default; `workdown serve --open` does. Matches `vite`, `next dev`, `cargo doc`, `mdbook serve`, etc. The CLI handles the launch via the `open` crate (not the server crate — `server` is library-shaped per ADR-006 and shouldn't pop browsers on its own). Mechanically: `server::bind()` returns the bound address (relevant when scan-on-conflict fired), CLI prints the URL, then calls `open::that()` if `--open` was passed, then `server::serve()` blocks. On `open` failure (devcontainer, SSH session, no browser available), log a warning + URL and keep serving — never fail the command. No headless-detection heuristics; try the call, report the outcome. Devcontainer dev flow uses VS Code's auto-forwarded-port notification instead of `--open`.
- **Distribution: binaries only via cargo-dist, no crates.io publishing.** All workspace crates set `publish = false`. Users install via cargo-dist installers (`irm get.workdown.sh | iex`, `curl ... | sh`, future package managers). No `cargo install workdown` path — workdown's audience is mostly non-Rust users; the niche of "Rust devs who reflexively `cargo install`" is small enough to defer. Easy to reverse later: switching to crates.io publishing would mean adding a `publish-jobs` step that builds `ui/dist/` then `cargo publish`es with `[package] include = ["ui/dist/**", ...]`. Avoiding that work now keeps the source tarball free of build artifacts and avoids the `cargo install --debug` edge case (rust-embed in debug mode reads from a `CARGO_MANIFEST_DIR` that no longer exists post-install).
- **Placeholder content: minimal page, complete plumbing.** The page itself is dead simple — `<h1>Workdown</h1>` plus a one-line "walking skeleton is alive" message. Everything *around* it is fully wired up so slice 2 doesn't have to backfill skeleton infrastructure before building real views. Plumbing in scope: root `+layout.svelte` setting default `<title>` via `<svelte:head>` and rendering children; root `+layout.ts` with `export const ssr = false` (true SPA mode); `app.html` with proper `lang`, viewport, charset, favicon link; `static/favicon.svg` (placeholder mark, replaceable later) and `static/robots.txt` (disallow all — local dev tool); `svelte.config.js` with `adapter-static` + `fallback: 'index.html'` + `outDir: '../ui/dist'`; `vite.config.ts` with `build.sourcemap: false` and a `/api/*` proxy stub targeting `localhost:3141` (forward-looking; no API yet); SvelteKit-strict `tsconfig.json`; `package.json` scripts (`dev`, `build`, `check`, `preview`); committed `ui/package-lock.json` (required by `npm ci`); empty `ui/dist/.gitkeep` so rust-embed's path resolves at compile time even on fresh clones; axum routes via a custom asset handler in `crates/server` (look up the path in the rust-embed; if found, serve with correct `Content-Type` from `mime_guess`; if missing AND the URL looks like an asset path with a non-`.html` extension, return 404; otherwise fall through to `index.html` for SPA routing) — chosen over the `axum-embed` crate to keep the routing logic explicit and dep-light; tokio runtime built inline in the serve handler (not `#[tokio::main]`) so other subcommands stay sync-cheap; xtask runs `npm ci` + `npm run check` + `npm run build` + `cargo build --release` (TS errors fail the release build); `.cargo/config.toml` alias; `.gitignore` covers `ui/node_modules/`, `ui/dist/`, `ui/.svelte-kit/`; `crates/core/defaults/config.yaml` gets a new top-level `serve:` section with `port: 3141` (sibling of `paths`, `schema`, `defaults`); CI gets a new "release pipeline" job that runs `cargo xtask build` on every PR (catches xtask / cargo-dist / SvelteKit-config breakage before release-tag push); both CI and `.github/build-setup.yml` pin node to version 20 via `setup-node`; README sections for "Installing" (cargo-dist installers only — no `cargo install workdown`), "Running locally", and "Working on the UI". Explicitly *not* in scope: any `/api/*` endpoints, version-string injection, SSE, real chrome (nav/sidebar/layout grid), CSS framework, UI tests, graceful Ctrl+C shutdown, request-tracing middleware — all deferred to later slices.
- **Web framework: axum.** Recorded explicitly. No realistic alternative considered: modern Rust default, well-supported, tower-compatible (lets us layer middleware cleanly when we want logging, compression, etc.), good tokio integration.

## Inputs the server handles

- CLI invocations (existing — unchanged)
- HTTP requests from the browser (new — page loads, asset loads, JSON API, SSE)
- File system events from a watcher (later slice — fires SSE pushes when `workdown-items/` changes)

## Scope (this skeleton)

- `workdown serve [--port N] [--open]` CLI command (defaults: auto-port, no browser open)
- `ui/` Svelte + TypeScript project initialized at the repo root, Vite builds to `ui/dist/`
- `cargo build --release` produces a working binary that contains the JS bundle
- `rust-embed` configured with `debug-embed = false`, embeds `ui/dist/` into the server crate
- `GET /` serves embedded `index.html`; `GET /assets/*` serves the bundle
- Startup log line: URL, port, PID
- Devcontainer: add node feature (`ghcr.io/devcontainers/features/node`)
- CI: add `actions/setup-node` + npm cache before the build step
- cargo-dist: produce the JS bundle once before the per-target matrix (mechanism TBD — see open questions)

## Acceptance

- Fresh clone → devcontainer rebuild → `cargo build --release --workspace` → `workdown serve` → browser shows a Svelte placeholder
- No node required at runtime (only at build time)
- Plain `cargo check` / `cargo test` in debug do not invoke node
- Dev workflow documented: `npm run dev` for UI iteration with HMR, `cargo build --release` for the embedded production binary

## Out of scope (for this skeleton — picked up in later slices)

- Real API endpoints — slice 2 (`first-view-end-to-end`)
- File watching / SSE live updates — slice 5 (`live-updates`)
- TLS, auth, remote serve — explicit non-goals of the milestone
